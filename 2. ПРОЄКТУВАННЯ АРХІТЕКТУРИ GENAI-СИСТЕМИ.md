



# 2. ПРОЄКТУВАННЯ АРХІТЕКТУРИ GENAI-СИСТЕМИ

## 2.1 Загальна концепція та клієнт-серверна архітектура

Система проєктується як вебдодаток із чітким розділенням на клієнтську (Frontend) та серверну (Backend) частини. Клієнтська частина реалізована на Vue.js з використанням D3.js і відповідає за взаємодію з користувачем, завантаження наборів даних у форматі CSV, відображення інтерактивних дашбордів та налаштування параметрів графіків. Використання D3.js на стороні клієнта дозволяє динамічно рендерити згенерований код безпосередньо в DOM-дереві без перезавантаження сторінки. Серверна частина виступає оркестратором: приймає табличні дані, виконує їх аналіз, формує промпти для мовних моделей, взаємодіє з LLM (через GPT API або локальні моделі) та виконує валідацію згенерованого коду.

## 2.2 Архітектурні вимоги та обмеження

Перед детальним описом компонентів системи необхідно зафіксувати функціональні та нефункціональні вимоги, а також врахувати фундаментальні обмеження, які накладає використання великих мовних моделей як основного компонента генерації коду.

**Функціональні вимоги** охоплюють: завантаження користувачем табличних даних у форматі CSV; автоматичне визначення структури набору даних; рекомендацію типу візуалізації на основі характеристик даних; генерацію виконуваного JavaScript-коду для бібліотеки D3.js; валідацію згенерованого коду перед його виконанням у клієнтському середовищі; ітеративне виправлення помилок у згенерованому коді; інтерактивне відображення дашборду в браузері.

**Нефункціональні вимоги** включають: *продуктивність* — час відгуку системи у межах кількох секунд для типових наборів даних; *приватність* — повний обсяг даних користувача не повинен передаватися до зовнішніх хмарних сервісів без явної згоди; *надійність* — система повинна коректно обробляти випадки, коли LLM не здатна згенерувати валідний код навіть після кількох ітерацій рефайнменту; *розширюваність* — архітектура повинна допускати заміну провайдера LLM без зміни решти компонентів.

**Обмеження великих мовних моделей** є відправною точкою для ключових архітектурних рішень. Сучасні дослідження [1, 14] виділяють три суттєвих обмеження, які враховано при проєктуванні:

1. *Обмежене контекстне вікно та неефективність обробки табличних даних.* Передача великих CSV-файлів безпосередньо до моделі неефективна як з погляду вартості токенів, так і з погляду якості. Дослідження TiInsight [1] показує, що LLM принципово погано працює з великими двовимірними структурами без спеціального абстрагування, а Table-GPT [14] демонструє, що “сирі” таблиці моделі схильні інтерпретувати одновимірно, плутаючи рядки і стовпці.
1. *Схильність до галюцинацій.* LLM можуть генерувати посилання на неіснуючі стовпці, неправильні типи даних або синтаксично коректний, але семантично хибний код. Це обґрунтовує необхідність окремого етапу валідації згенерованого коду перед його виконанням (підрозділ 2.6).
1. *Стохастичність генерації.* Навіть за однакового вхідного промпту модель може повертати різні результати, що ускладнює як відтворюваність експериментів, так і практичне використання системи. Для компенсації цієї варіативності архітектура передбачає rule-based базовий шар (підрозділ 2.4) та ітеративний цикл валідації (підрозділ 2.6).

## 2.3 Модуль попередньої обробки та абстрагування даних

### 2.3.1 Алгоритм визначення схеми (Schema Inference — визначення схеми)

Модуль *Schema Inference* виконує автоматичний аналіз завантаженого CSV-файлу та формує структурований опис набору даних, який згодом використовується як вхід для модуля рекомендації типу візуалізації та для конструювання промпту LLM. Алгоритм виконує такі кроки: парсинг CSV з визначенням розділювача і наявності рядка заголовків; визначення типу кожного стовпця на основі евристичного аналізу значень (числовий, категоріальний, часовий, текстовий); обчислення статистичних характеристик (мінімум, максимум, кардинальність, часовий діапазон, частка пропущених значень); формування метаданих у вигляді структурованого JSON-об’єкта, який містить назви стовпців, їхні типи, статистичні характеристики та загальну кількість рядків.

Результат роботи модуля — компактний опис набору даних розміром у кілька сотень токенів незалежно від реального обсягу CSV. Концептуально такий підхід відповідає ідеї *Hierarchical Data Context* (HDC), запропонованій у роботі TiInsight [1, 2], де схема витягується з великих реляційних баз даних через розподілену обробку за схемою Map-Reduce.

### 2.3.2 Стратегії передачі даних: Full CSV проти Schema + Sample

Одне з ключових архітектурних рішень системи стосується способу передачі інформації про дані до мовної моделі. У літературі з NL2Vis виділяють два принципово різних підходи [1, 14]:

**Підхід Full CSV** передбачає включення всього (або значної частини) набору даних безпосередньо в текст промпту. Перевагою є повнота інформації, доступної моделі; недоліками — швидке вичерпання контекстного вікна, висока вартість запиту, зниження якості генерації через “розмивання” уваги моделі на великому обсязі однотипного тексту [14].

**Підхід Schema + Sample** передбачає передачу до моделі лише структурного опису набору даних (отриманого від модуля *Schema Inference*) та невеликої репрезентативної вибірки з 1–3 рядків. Цей підхід радикально зменшує обсяг переданих токенів і дозволяє моделі сконцентруватися на типах і ролях стовпців. Дослідження TiInsight [1] показує, що для задачі генерації коду візуалізації такий підхід забезпечує порівнянну або кращу якість порівняно з Full CSV при значно меншій вартості.

У спроєктованій системі **Schema + Sample є дефолтною стратегією** з таких причин: відповідність вимозі приватності; економія токенів і вартості запитів; можливість роботи з великими наборами даних, які принципово не вмістилися б у контекстне вікно. Водночас архітектура передбачає **підтримку обох стратегій** через єдиний інтерфейс модуля підготовки даних — це необхідно для експериментального порівняння двох підходів у Розділі 5.

### 2.3.3 Локалізація даних та приватність

Принципово важливим аспектом архітектури є розділення двох потоків даних. До LLM передається лише структурна схема та невелика вибірка (за замовчуванням — 3 рядки), тобто користувач контролює, який саме обсяг його даних залишає локальне середовище. Повний обсяг CSV залишається на стороні клієнта (у пам’яті браузера) і використовується виключно для фінального рендерингу згенерованим D3.js кодом. Таке розділення забезпечує сумісність із вимогами обробки персональних даних і робить можливим використання системи в сценаріях, де передача повних даних до зовнішніх сервісів є неприйнятною (медичні, фінансові, корпоративні дані).

## 2.4 Модуль вибору типу візуалізації

У літературі з NL2Vis існують два основних підходи до вибору типу графіка: повністю rule-based (правила відображення типів стовпців на типи графіків) та повністю LLM-based. Перший забезпечує передбачуваність і нульову вартість інференсу, але погано справляється з неоднозначними випадками; другий є гнучкішим, але менш передбачуваним. У спроєктованій системі застосовано **гібридний підхід**, концептуально близький до алгоритму TiChart у роботі TiInsight [2], який поєднує переваги обох методів.

**Етап 1: Rule-based базовий рівень.** На основі метаданих від модуля *Schema Inference* система застосовує таблицю правил відображення (таблиця 2.1).

*Таблиця 2.1 — Правила відображення типів стовпців на типи візуалізації*

|Комбінація стовпців                         |Рекомендований тип графіка          |
|--------------------------------------------|------------------------------------|
|Один часовий + один числовий                |Лінійний графік (line chart)        |
|Один категоріальний + один числовий         |Стовпчикова діаграма (bar chart)    |
|Два числових                                |Діаграма розсіювання (scatter plot) |
|Один категоріальний з низькою кардинальністю|Кругова діаграма (pie chart)        |
|Два категоріальних + один числовий          |Згрупована стовпчикова діаграма     |
|Один часовий + декілька числових            |Мультилінійний графік               |
|Два числових + один категоріальний          |Scatter plot з кольоровим кодуванням|

**Етап 2: LLM-уточнення для неоднозначних випадків.** Якщо rule-based алгоритм не знаходить однозначної відповідності, система делегує рішення мовній моделі. LLM отримує не повні дані, а лише схему (підрозділ 2.3.2) та список кандидатних типів графіків і має обрати найдоречніший у структурованому форматі відповіді (JSON).

Такий гібридний підхід забезпечує детермінованість для типових випадків, гнучкість для складних, економію ресурсів за рахунок виклику LLM лише за потреби, а також прозорість — система може пояснити, на основі якого правила або обґрунтування було обрано конкретний тип візуалізації. Важливо, що результат роботи цього модуля — рекомендація, а не остаточне рішення: користувач може перевизначити її через інтерфейс.

## 2.5 Модуль кодогенерації

### 2.5.1 Патерн Visualization Generator (Генератор візуалізацій)

Ключовим архітектурним принципом модуля кодогенерації є застосування патерну *Visualization Generator*, описаного у каталозі промпт-патернів White et al. [27, 28, 29]: LLM використовується не як рендеринг-інструмент, а як генератор виконуваного коду для спеціалізованої бібліотеки візуалізації. Такий підхід дозволяє використовувати сильні сторони обох компонентів — здатність LLM до узагальнення та знання типових патернів візуалізаційного коду, а також потужність D3.js у побудові інтерактивних графіків з точним контролем над DOM-деревом. Архітектурний аналог такого підходу для генерації саме D3.js візуалізацій представлено в системі InterChat [5, 6], де LLM-агент відповідає за генерацію коду прив’язки даних до DOM-елементів.

Вхідними даними для модуля є структурована схема, рекомендований тип візуалізації, вибірка з 1–3 рядків та опціональні налаштування користувача. На виході модуль повертає JavaScript-функцію, яка приймає повний масив даних як аргумент і будує D3.js візуалізацію в заданому контейнері DOM.

### 2.5.2 Абстракція над LLM-провайдером

Архітектура системи передбачає **LLM-агностичний інтерфейс** — окремий шар, який приховує деталі взаємодії з конкретним провайдером моделі за єдиним API з трьома основними методами: `generateCode`, `refineCode`, `selectChartType`. Це дозволяє підключати як хмарні сервіси, так і локальні моделі без зміни решти компонентів.

*Хмарні API моделі* (GPT-4, GPT-4o) забезпечують найвищу якість генерації коду, мають великі контекстні вікна та не потребують локальних обчислювальних ресурсів. Недоліками є вартість запитів, залежність від доступу до мережі та необхідність передачі метаданих до зовнішнього сервісу. Ці моделі обрано як базовий провайдер для експериментів.

*Локальні моделі* [31, 32] запускаються на машині користувача, забезпечуючи повну приватність даних та незалежність від зовнішніх сервісів. Обмеження експериментального середовища (8 ГБ оперативної пам’яті без виділеного GPU) робить придатними лише квантовані моделі з розміром 1.5–3 мільярди параметрів. До кандидатів належать Qwen2.5-Coder-3B, DeepSeek-Coder-1.3B, CodeGemma-2B та Phi-3.5-mini — у квантизації Q4, запуск через середовище llama.cpp або Ollama. Вибір конкретної моделі для фінальних експериментів обґрунтовується в Розділі 5 на основі попереднього порівняльного тестування.

### 2.5.3 Конструювання промптів

Промпт формується оркестратором як комбінація системної інструкції, структурної схеми та вибірки даних, рекомендованого типу графіка, опціональних few-shot прикладів та формату очікуваної відповіді. Детальний опис стратегій промптингу (zero-shot, few-shot, chain-of-thought [30]) та конкретні шаблони наведено в Розділі 3.

## 2.6 Модуль валідації та ітеративного рефайнменту

Згенерований мовною моделлю код не можна виконувати в клієнтському середовищі без попередньої перевірки з двох причин: через обмеження LLM, описані в підрозділі 2.2; через загрозу безпеки при безпосередньому виконанні довільного JavaScript-коду. Необхідність такої валідації підтверджується в дослідженні LEVER [20, 21], де показано, що перевірка коду через його реальне виконання суттєво покращує метрики успішності генерації.

### 2.6.1 Статичний аналіз

Перший рівень — статичний аналіз коду без його виконання: синтаксична коректність через парсинг AST (abstract syntax tree) за допомогою Esprima або Acorn; перевірка за білим списком дозволених API (`d3`, аргумент з даними, DOM-контейнер); блокування небезпечних глобальних об’єктів (`eval`, `Function`, `fetch`, `XMLHttpRequest`, доступ до `document.cookie`); структурна відповідність очікуваній сигнатурі функції. Статичний аналіз дозволяє відкинути очевидно некоректні генерації без витрачання ресурсів на запуск у пісочниці.

### 2.6.2 Виконання в ізольованому середовищі (Sandbox — пісочниця)

Код, що пройшов статичний аналіз, виконується в ізольованому середовищі на сервері за допомогою модуля `vm` Node.js з обмеженим контекстом: мок-реалізацією D3.js для тестового рендерингу та схемою і вибіркою даних. Виконання обмежується за часом та використанням пам’яті. Архітектура *execution feedback* концептуально відповідає підходу OpenCodeInterpreter [15, 16], де діагностичні повідомлення середовища виконання використовуються для ітеративного вдосконалення коду.

### 2.6.3 Цикл Self-Refine (ітеративний рефайнмент)

Якщо на етапі статичного аналізу або виконання в пісочниці виявлено помилку, система не відхиляє результат одразу, а запускає цикл ітеративного виправлення. Концептуальною основою цього циклу є метод *Self-Refine* [24], у якому одна й та сама LLM ітеративно покращує власний результат на основі згенерованого зворотного зв’язку без додаткового тренування. Близька за суттю ідея *Self-Debug* [22, 23] описує, як модель, отримавши повідомлення про помилку та stack trace, здатна самостійно виправити згенерований код.

Алгоритм циклу включає: структуроване формування повідомлення про помилку; побудову нового промпту з оригінальним кодом, текстом помилки та інструкцією виправити код зі збереженням початкового наміру; повторний статичний аналіз і виконання в пісочниці; повторення до успішного виконання або досягнення ліміту ітерацій (за замовчуванням — 3). Архітектурно близький підхід реалізовано у VisPath [7, 8], де паралельно генеруються декілька варіантів коду з подальшим вибором кращого на основі фідбеку.

**Обробка вичерпання спроб.** Якщо після трьох ітерацій код не проходить валідацію, система переходить у *fallback-режим*: генерує просту візуалізацію за заздалегідь підготованим шаблоном, що відповідає рекомендованому типу графіка. Це забезпечує виконання нефункціональної вимоги надійності — користувач завжди отримує результат, навіть якщо він не оптимальний. Факт спрацьовування fallback-механізму логується і враховується в експериментальній оцінці якості системи (Розділ 5).

## 2.7 Мультиагентна декомпозиція системи

Описані вище модулі природно лягають на парадигму **мультиагентної архітектури**, яка в сучасних NL2Vis-системах є домінуючим підходом. Системи PlotGen [3, 4], InterChat [5, 6] та AuraGenome [9, 10] демонструють ефективність декомпозиції задачі генерації візуалізацій на спеціалізованих агентів зі вузькими зонами відповідальності. На відміну від “сильних” мультиагентних систем з автономним плануванням та динамічним вибором інструментів, запропонована архітектура використовує *фіксовану топологію взаємодії*, що забезпечує передбачуваність, відтворюваність експериментів та простоту відлагодження — властивості, критично важливі для наукового дослідження.

У системі виділено чотири агенти:

**Schema Analyst Agent** (rule-based, без LLM) відповідає за аналіз завантаженого CSV та формування структурованого опису схеми. Реалізація без мовної моделі обґрунтована тим, що задача визначення типів стовпців надійно вирішується евристичними алгоритмами.

**Chart Type Recommender Agent** (гібридний: rule-based + LLM) обирає тип візуалізації на основі метаданих, використовуючи таблицю правил (підрозділ 2.4) як базовий рівень і делегуючи рішення LLM лише для неоднозначних випадків.

**Code Generator Agent** (LLM) є основним генеративним агентом системи. Отримує схему, вибірку і рекомендований тип графіка, повертає код D3.js-візуалізації. Архітектурно близький за функцією до агента Avis у системі InterChat [5].

**Code Validator & Debugger Agent** (LLM + статичний аналіз) відповідає за валідацію та ітеративне виправлення коду в рамках Self-Refine циклу. Ключова особливість — окремий промпт, оптимізований під задачу виправлення помилок, а не генерації з нуля. Розділення ролей генератора та дебагера відповідає принципам, описаним у PlotGen [3, 4] та AuraGenome [9, 10], та підвищує якість виправлень порівняно з підходом, коли той самий промпт використовується для обох задач.

Координація агентів здійснюється **оркестратором** — центральним компонентом на серверній стороні, який послідовно викликає агентів, передає між ними дані, обробляє помилки та приймає рішення про перехід у fallback-режим. Оркестратор також відповідає за логування усіх взаємодій для подальшого експериментального аналізу.

## 2.8 Покроковий workflow з обробкою виняткових ситуацій

Повний процес генерації дашборду зображено на рисунку 2.2 (діаграма послідовності). Основний потік виконання:

1. Користувач завантажує CSV-файл через інтерфейс Vue.js.
1. Frontend передає файл на Backend, зберігає повний обсяг даних у пам’яті браузера.
1. *Schema Analyst Agent* аналізує CSV та формує структурований опис схеми.
1. *Chart Type Recommender Agent* обирає тип візуалізації.
1. *Code Generator Agent* формує D3.js код на основі схеми, вибірки та рекомендованого типу.
1. *Code Validator Agent* виконує статичний аналіз та sandbox-виконання.
1. У разі успіху код повертається на Frontend; у разі помилки запускається Self-Refine цикл.
1. Frontend монтує валідований код у DOM та виконує його з повним набором даних, отримуючи інтерактивний графік.

**Обробка виняткових ситуацій.** Архітектура передбачає коректну реакцію на: зламаний або порожній CSV (модуль Schema Analyst повертає помилку парсингу без звернень до LLM); неоднозначну схему (система повідомляє користувача та пропонує ручний вибір типу графіка); помилку виконання коду після 3 ітерацій Self-Refine (активується fallback-режим); таймаут або недоступність API (повторні запити з експоненційною затримкою, переключення на локальну модель за її наявності); некоректну відповідь LLM (обробляється як помилка валідації). Така обробка забезпечує виконання вимоги надійності та відповідає принципу *graceful degradation* — система продовжує надавати корисний результат навіть за часткової відмови окремих компонентів.

## 2.9 Висновки до розділу

У цьому розділі запропоновано архітектуру GenAI-системи автоматичної генерації інтерактивних дашбордів, яка реалізує клієнт-серверну модель з мультиагентною декомпозицією на серверній стороні. Ключовими архітектурними рішеннями є: застосування підходу Schema + Sample замість передачі повного обсягу даних до LLM, що забезпечує приватність, економію токенів та масштабованість; гібридний підхід до вибору типу візуалізації, який поєднує надійність rule-based правил з гнучкістю LLM; патерн *Visualization Generator*, що використовує LLM як кодогенератор, а не як рендеринг-інструмент; LLM-агностичний інтерфейс, що підтримує як хмарні API, так і локальні квантовані моделі; ітеративний цикл *Self-Refine* з fallback-механізмом, що забезпечує надійність; декомпозиція на чотири спеціалізовані агенти з єдиним оркестратором. Запропонована архітектура створює основу для детальної розробки алгоритмів генерації (Розділ 3), прототипної реалізації (Розділ 4) та експериментального порівняння стратегій передачі даних і класів моделей (Розділ 5).
