У даному розділі формалізовано алгоритми чотирьох ключових етапів системи, архітектура яких описана в Розділі 2: виведення схеми даних (Schema Inference), вибір типу візуалізації на базі правил, конструювання промптів (Prompt Engineering) та ітеративний рефайнмент коду через ізольоване виконання. ## 3.1 Алгоритм визначення схеми та підготовки даних (Schema Inference)

Архітектурне обґрунтування підходу Schema + Sample та його переваги над Full CSV наведено в підрозділі 2.3. Концептуально підхід відповідає ідеї Hierarchical Data Context (HDC), запропонованій у роботі TiInsight [1, 2], де схема витягується з великих реляційних баз даних через розподілену обробку. У даному підрозділі формалізовано конкретні кроки алгоритму, критерії класифікації стовпців та структуру вихідного артефакту.

**Крок 1. Парсинг та нормалізація.** Алгоритм зчитує перший рядок CSV як масив назв стовпців $C = {c_1, c_2, \ldots, c_k}$ та перші $N = 100$ рядків як матрицю значень $V$. Автоматично визначається роздільник (кома, крапка з комою, табуляція) за частотою входження символів у першому рядку.

**Крок 2. Визначення типу стовпця (Type Inference).** Сучасні дослідження [14, 15] демонструють, що LLM схильні інтерпретувати “сирі” таблиці одновимірно, плутаючи рядки і стовпці, тому класифікація типів виконується детерміновано, до будь-якого звернення до мовної моделі. Для кожного стовпця $c_i$ застосовується ієрархія евристичних правил:

$$\text{type}(c_i) = \begin{cases} \texttt{Temporal} & \text{якщо} \geq 80% \text{ значень відповідають ISO 8601 або } dd/mm/yyyy \ \texttt{Numeric} & \text{якщо} \geq 90% \text{ значень парсяться як } \mathbb{R} \ \texttt{Categorical} & \text{інакше} \end{cases}$$

Порогові значення 80% та 90% залишають простір для пропусків і некоректних записів без хибної класифікації.

**Крок 3. Обчислення статистичних характеристик.** Залежно від визначеного типу обчислюється:

- `Numeric`: min, max, середнє, стандартне відхилення
- `Categorical`: кардинальність (кількість унікальних значень), мода
- `Temporal`: мінімальна та максимальна дата, крок (день/місяць/рік)

Додатково для кожного стовпця обчислюється частка пропущених значень $\text{null_rate} = |\text{порожні}| / N$.

**Крок 4. Формування вибірки.** З матриці $V$ випадково обирається $s = 3$ рядки (стратифіковано, якщо наявний стовпець `Categorical`). Наявність реальних прикладів значень допомагає мовній моделі краще зрозуміти семантику даних [1, 2].

**Вихідний артефакт.** Результатом є JSON-об’єкт фіксованої структури (Лістинг 3.1). Розмір цього об’єкта становить ~200–400 токенів незалежно від обсягу вхідного CSV, що є ключовою передумовою масштабованості системи та відповідності вимозі приватності (підрозділ 2.3.3).

**Лістинг 3.1 — Структура схеми (Schema JSON)**

```json
{
  "total_rows": 1500,
  "columns": [
    {
      "name": "date",
      "type": "Temporal",
      "range": ["2023-01-01", "2023-12-31"],
      "step": "month",
      "null_rate": 0.0
    },
    {
      "name": "revenue",
      "type": "Numeric",
      "min": 1200,
      "max": 98400,
      "mean": 45300,
      "null_rate": 0.02
    },
    {
      "name": "region",
      "type": "Categorical",
      "cardinality": 4,
      "mode": "North",
      "null_rate": 0.0
    }
  ],
  "sample": [
    {"date": "2023-03-01", "revenue": 41200, "region": "North"},
    {"date": "2023-07-01", "revenue": 67800, "region": "South"},
    {"date": "2023-11-01", "revenue": 23100, "region": "East"}
  ]
}
```

## 3.2 Алгоритм вибору типу візуалізації на базі правил

Гібридна архітектура модуля (rule-based + LLM для неоднозначних випадків), концептуально близька до алгоритму TiChart [1, 2], обґрунтована в підрозділі 2.4. Дослідження сучасних NL2Vis-систем доводять, що користувачі здебільшого віддають перевагу базовим візуалізаціям, для яких існують чіткі евристичні правила [35], тому покладання виключно на LLM у виборі типу графіка є надлишково ресурсомістким. У даному підрозділі формалізовано повну таблицю правил та логіку прийняття рішення.

**Вхід:** схема $S$ від алгоритму 3.1 — множина стовпців із типами та характеристиками.

**Позначення:**

- $N_T, N_C, N_N$ — кількість стовпців типів `Temporal`, `Categorical`, `Numeric` відповідно
- $\kappa(c)$ — кардинальність категоріального стовпця $c$
- $\kappa_{\text{low}} \leq 6$ — поріг низької кардинальності

**Таблиця 3.1 — Правила відображення типів стовпців на типи візуалізації (пріоритет зверху вниз)**

|Умова                                                   |Рекомендований тип                  |
|--------------------------------------------------------|------------------------------------|
|$N_T = 1,\ N_N \geq 2$                                  |Мультилінійний графік               |
|$N_T = 1,\ N_N = 1$                                     |Лінійний графік                     |
|$N_C = 2,\ N_N = 1$                                     |Згрупована стовпчикова діаграма     |
|$N_C = 1,\ N_N = 1$                                     |Стовпчикова діаграма                |
|$N_C = 1,\ N_N = 2$                                     |Scatter plot з кольоровим кодуванням|
|$N_N = 2$                                               |Scatter plot                        |
|$N_C = 1,\ N_N = 0,\ \kappa(c) \leq \kappa_{\text{low}}$|Кругова діаграма                    |
|інакше                                                  |→ делегування LLM                   |

**Алгоритм (псевдокод):**

```
function selectChartType(schema S):
    підрахувати N_T, N_C, N_N із S.columns

    for кожне правило в таблиці 3.1 (за пріоритетом):
        if умова виконана:
            return (chartType, "rule-based", ID_правила)

    // Неоднозначний випадок — делегування LLM
    candidates ← ["bar", "line", "scatter", "pie", "heatmap"]
    prompt ← buildChartSelectPrompt(S, candidates)
    response ← llm.call(prompt, format="JSON")
    return (response.chartType, "llm", response.reasoning)
```

**Вихід:** кортеж `(chartType, source, reasoning)`, де `source` ∈ `{"rule-based", "llm"}` фіксується для експериментального аналізу в Розділі 5. Результат передається до модуля кодогенерації як жорстке обмеження в системному промпті, що унеможливлює вибір LLM альтернативного типу графіка на етапі генерації коду та нівелює ризик галюцинацій [14, 16].

## 3.3 Проєктування промптів (Prompt Engineering) для генерації D3.js

Патерн Visualization Generator та обґрунтування використання CoT описані в підрозділах 2.5.1 та 2.5.3. Оскільки бібліотека D3.js має низькорівневий синтаксис та вимагає точних маніпуляцій з DOM-деревом [5, 6], генерація коду потребує суворо структурованого підходу. У даному підрозділі формалізовано конкретну структуру промптів та механізм їх динамічного конструювання.

**Архітектура промпту.** Системний промпт будується оркестратором як конкатенація п’яти блоків:

$$P = B_{\text{role}} \oplus B_{\text{schema}} \oplus B_{\text{chart}} \oplus B_{\text{shots}} \oplus B_{\text{constraints}}$$

де $\oplus$ — послідовне об’єднання, $B_{\text{shots}}$ включається лише у few-shot режимі.

**Блок $B_{\text{role}}$ — визначення ролі та завдання:**

```text
You are an expert D3.js v7 developer.
Your task is to generate a JavaScript function:
  renderChart(data, containerSelector)
that creates an interactive {chartType} visualization.
Think step by step:
  1. Identify which columns map to axes/encoding channels
  2. Define scales and domains
  3. Bind data to DOM elements
  4. Add axes, labels, legend, and tooltip
```

Інструкція “Think step by step” активує Chain-of-Thought міркування [30], що підвищує структурну коректність згенерованого коду.

**Блок $B_{\text{schema}}$ — контекст даних:**

```text
[Dataset]
Schema: {SchemaJSON}
Sample: {SampleJSON}
```

Сюди підставляється JSON-артефакт алгоритму 3.1. Розмір блоку — 200–400 токенів незалежно від обсягу вхідного CSV.

**Блок $B_{\text{chart}}$ — обмеження типу:**

```text
[Chart Specification]
Chart type: {chartType}  (determined by rule-based algorithm, do not change)
X-axis: {temporal/categorical column name}
Y-axis: {numeric column name}
```

Явне зазначення стовпців усуває клас галюцинацій, пов’язаних із некоректним маппінгом полів [14, 16].

**Блок $B_{\text{shots}}$ — few-shot приклади (опціонально).** За наявності у сховищі шаблону для заданого `chartType` додається один завершений приклад пари (схема → код). Дослідження Li et al. [35] показує, що few-shot приклади знижують кількість синтаксичних помилок D3.js v7 API, зокрема некоректне використання застарілих методів (`d3.nest`, `d3.csv` з колбеком).

**Блок $B_{\text{constraints}}$ — технічні обмеження:**

```text
[Constraints]
- Use ONLY D3.js v7. Forbidden: d3.nest(), d3.csv(), d3.tsv()
- Calculate width/height dynamically from containerSelector bounds
- Append SVG to containerSelector, do not use hardcoded IDs
- Include: axis labels, legend, hover tooltip via d3.select().on("mouseover")
- Return ONLY valid JavaScript inside a single ```javascript block
- Do not output HTML, CSS, or explanation text
```

Відповідно до патерну Visualization Generator [27, 28, 29], модель отримує заборону на спроби самостійно “відобразити” дані — натомість вона генерує виконуваний код для спеціалізованого рендерера.

**Режими генерації.** Система підтримує три режими, які порівнюються експериментально в Розділі 5:

|Режим    |Склад промпту                                                                                   |Очікуваний ефект    |
|---------|------------------------------------------------------------------------------------------------|--------------------|
|Zero-shot|$B_{\text{role}} \oplus B_{\text{schema}} \oplus B_{\text{chart}} \oplus B_{\text{constraints}}$|Базова якість       |
|Few-shot |+ $B_{\text{shots}}$ (1 приклад)                                                                |Менше помилок API   |
|CoT      |$B_{\text{role}}$ з покроковою інструкцією                                                      |Краща структура коду|

**Постобробка відповіді.** Після отримання відповіді від LLM оркестратор екстрагує код регулярним виразом ````javascript ... ````, відкидаючи будь-який текст поза блоком. Відсутність блоку трактується як помилка генерації та одразу запускає цикл рефайнменту (підрозділ 3.4).

## 3.4 Алгоритм валідації та ітеративного рефайнменту (Self-Refine)

Концептуальна основа (Self-Refine [24], Self-Debug [22, 23], LEVER [20, 21]) та архітектурне обґрунтування наведені в підрозділі 2.6. Дослідження LEVER [20, 21] показує, що перевірка коду через його реальне виконання суттєво покращує метрики успішності генерації. У даному підрозділі формалізовано повний алгоритм із критеріями переходу між етапами.

**Загальна схема.** Валідація складається з двох послідовних рівнів; провал будь-якого запускає цикл рефайнменту:

$$\text{код} \xrightarrow{\text{Рівень 1: статичний аналіз}} \xrightarrow{\text{Рівень 2: sandbox}} \text{успіх} \mid \text{помилка} \rightarrow \text{refinement}$$

**Рівень 1. Статичний аналіз (без виконання).**

Перевірки виконуються послідовно; перша невдала зупиняє аналіз:

```
function staticAnalyze(code):
    ast ← parse(code)          // Acorn/Esprima; SyntaxError → відхилити

    // Перевірка сигнатури
    fn ← findFunction(ast, "renderChart")
    if fn.params ≠ ["data", "containerSelector"]:
        return Error("wrong_signature")

    // Білий список API
    calls ← extractAllCallExpressions(ast)
    forbidden ← ["eval", "Function", "fetch", "XMLHttpRequest",
                  "document.cookie", "d3.nest", "d3.csv", "d3.tsv"]
    if any(call ∈ forbidden for call in calls):
        return Error("forbidden_api", call)

    // Структурна перевірка D3
    if not contains(ast, "d3.select"):
        return Error("missing_d3_select")

    return OK
```

Статичний аналіз відхиляє очевидно некоректні генерації без витрат на запуск пісочниці.

**Рівень 2. Виконання в ізольованому середовищі (Sandbox).**

Архітектура execution feedback концептуально відповідає підходу, описаному в роботах з верифікації коду через виконання [20, 21], де діагностичні повідомлення середовища використовуються для ітеративного вдосконалення:

```
function sandboxExecute(code, schema, sample):
    context ← {
        d3: MockD3(),          // мок-об'єкт з логуванням викликів
        data: sample,          // 3 рядки з Schema JSON
        container: "#test-div"
    }

    result ← vm.runInContext(code, context,
                             timeout=3000ms,
                             memoryLimit=64MB)

    // Перевірка фактичного використання даних
    if MockD3.dataCalls.count == 0:
        return Error("data_not_bound")

    if MockD3.svgAppended == false:
        return Error("no_svg_appended")

    return OK | RuntimeError(stack_trace)
```

MockD3 імітує D3.js API (`.select`, `.append`, `.data`, `.enter`, `.attr`, `.scale*`), перехоплює виклики та перевіряє коректність data binding [5, 6].

**Цикл рефайнменту.** Методи Self-Refine [24] та Self-Debug [22, 23] доводять, що LLM здатна самостійно виправляти власні помилки, якщо їй надати текст помилки виконання (stack trace). На цьому принципі побудовано цикл:

```
function refine(code, error, schema, chartType):
    i ← 0
    while i < MAX_ITERATIONS (=3):
        feedback ← buildFeedbackPrompt(code, error)
        code ← llm.call(feedback)
        code ← extractCodeBlock(code)

        result ← staticAnalyze(code)
        if result == OK:
            result ← sandboxExecute(code, schema, sample)

        if result == OK:
            return (code, SUCCESS, iterations=i+1)

        error ← result.error
        i ← i + 1

    // Вичерпано ліміт
    return (fallbackTemplate(chartType), FALLBACK, iterations=3)
```

**Лістинг 3.2 — Шаблон промпту зворотного зв’язку $P_{\text{refine}}$**

```text
[Error Report]
Iteration: {i} / 3
Error type: {error_type}
Stack trace:
{error_trace}

[Original Code]
{code}

[Task]
Fix the error above. Preserve the original chart type ({chartType})
and data mapping. Return ONLY the corrected ```javascript block.
```

Ліміт у 3 ітерації відповідає балансу між надійністю та вартістю запитів. Після вичерпання ліміту активується fallback-механізм із заздалегідь підготованим статичним шаблоном D3.js для заданого типу графіка [7, 8]. Використання виконання коду як механізму зворотного зв’язку дозволяє суттєво підвищити надійність системи без необхідності донавчання (fine-tuning) моделей [24].

**Вихідний артефакт.** Після успішного проходження обох рівнів оркестратор повертає на Frontend структуру:

```json
{
  "code": "function renderChart(data, containerSelector) { ... }",
  "status": "success | fallback",
  "iterations": 0,
  "chart_type": "line",
  "validation_log": [...]
}
```

Поля `status`, `iterations` та `validation_log` логуються для експериментального аналізу надійності системи в Розділі 5.

## 3.5 Висновки до розділу

У даному розділі формалізовано алгоритмічну основу чотирьох модулів системи. Алгоритм Schema Inference забезпечує детерміновану класифікацію стовпців за ієрархією евристичних правил та формує компактний JSON-артефакт (~200–400 токенів), що є спільним вхідним форматом для всіх наступних модулів [1, 2]. Rule-based алгоритм вибору типу візуалізації охоплює сім типових комбінацій стовпців із чітким пріоритетом і делегує LLM лише неоднозначні випадки, забезпечуючи детермінованість для переважної більшості запитів [35]. Триблоковий механізм конструювання промптів (zero-shot / few-shot / CoT) дозволяє варіювати стратегію генерації без зміни архітектури відповідно до патерну Visualization Generator [27, 28, 29] та принципів CoT prompting [30], що створює основу для порівняльних експериментів у Розділі 5. Дворівневий алгоритм валідації (статичний аналіз + sandbox) у поєднанні з Self-Refine циклом [24] та методом Self-Debug [22, 23] (до 3 ітерацій) і fallback-механізмом забезпечує виконання нефункціональної вимоги надійності — система завжди повертає результат навіть за часткової відмови генеративного компонента.
